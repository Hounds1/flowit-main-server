== 작업 알림 WebSocket

[cols="1,4",role="endpoint-summary"]
|===
|상위 문서 |link:websocket-notifications.html[알림 WebSocket]
|구독 destination |``/user/queue/notifications``
|payload |``NotificationAlertResponse``
|도메인 |작업 생성 및 변경
|역할 |작업 생성, 상태 변경, 담당자 할당/해제 이벤트를 개인 알림으로 전달합니다.
|주의 |별도 destination이 아니라 공통 알림 큐로 내려오는 작업 도메인 payload 세부 계약입니다.
|===

== 발송 케이스

[cols="1,1,4",options="header",role="api-table"]
|===
|알림 유형 |수신자 |설명

|``TASK_CREATED``
|작업 생성자를 제외한 지정 담당자
|새 작업이 생성되었음을 지정 담당자에게 알립니다. 생성 시 담당자가 지정되면 같은 작업에 대해 ``TASK_ASSIGNED``가 이어서 내려올 수 있습니다.

|``TASK_DATE_CHANGED``
|현재 미발행
|작업 시작 예정일 또는 마감 예정일 변경 알림을 위한 예약 타입입니다. 현재는 알림 기록과 WebSocket payload를 생성하지 않습니다.

|``TASK_STATUS_CHANGED``
|변경자를 제외한 작업 생성자 + 현재 담당자
|작업 상태가 변경되었음을 알립니다.

|``TASK_PROGRESS_CHANGED``
|현재 미발행
|작업 진행도 변경 알림을 위한 예약 타입입니다. 현재는 알림 기록과 WebSocket payload를 생성하지 않습니다.

|``TASK_ASSIGNED``
|새 담당자
|현재 사용자가 작업 담당자로 할당되었음을 알립니다.

|``TASK_UNASSIGNED``
|직전 담당자
|현재 사용자가 작업 담당자에서 해제되었음을 알립니다.
|===

== 수신자 계산 기준

[cols="1,4",role="endpoint-summary"]
|===
|기본 범위 |작업 생성자와 현재 담당자를 후보로 사용합니다.
|변경자 제외 |알림을 발생시킨 사용자는 수신자에서 제외합니다.
|중복 제거 |작업 생성자와 현재 담당자가 같은 사용자이면 한 번만 발송합니다.
|담당자 변경 |담당자 변경이 포함되면 직전 담당자에게 ``TASK_UNASSIGNED``, 현재 담당자에게 ``TASK_ASSIGNED``를 각각 발송합니다.
|생성 시 담당자 |작업 생성과 동시에 담당자가 지정되면 지정 담당자에게 ``TASK_CREATED`` 이후 ``TASK_ASSIGNED``가 순차 발송될 수 있습니다.
|직접 알림 우선 |생성 시 담당자 지정 케이스를 제외하면, ``TASK_ASSIGNED`` 또는 ``TASK_UNASSIGNED``를 받은 사용자는 같은 이벤트의 일반 알림 수신자에서 제외해 중복 알림을 막습니다.
|빈 수신자 |변경자 제외와 중복 제거 이후 수신자가 없으면 알림을 생성하지 않습니다.
|===

== 발송 순서와 그룹 처리

서버는 같은 활동 레코드에서 파생된 작업 알림을 하나의 내부 그룹으로 다룹니다.
그룹 식별 값과 그룹 순서 값은 WebSocket payload 또는 목록 API 응답에 노출되지 않습니다.
클라이언트는 내려온 ``NotificationAlertResponse``를 개별 알림으로 렌더링하되, 같은 수신자에게 도착한 순서를 유지하면 됩니다.

다중 서버 인스턴스 환경에서도 서버는 수신자별 미발송 알림 스트림을 직렬 발송합니다.
따라서 같은 사용자에게 두 작업 생성 이벤트가 거의 동시에 발생해도 각 작업 그룹 내부의 ``TASK_CREATED`` -> ``TASK_ASSIGNED`` 흐름은 서버 발송 순서에서 끊기지 않도록 처리됩니다.
단, WebSocket은 클라이언트 수신 완료를 확인하는 채널이 아니므로 재접속 후에는 link:notifications-list.html[알림 목록 조회 API]로 보정해야 합니다.

== payload 해석 기준

[cols="1,1,4",options="header",role="api-table"]
|===
|필드 |의미 |설명

|``scope.id``
|``workspaceId``
|알림이 발생한 워크스페이스 식별자입니다.

|``actor.id``
|``userId``
|알림을 발생시킨 사용자 식별자입니다.

|``subject.id``
|``taskId``
|알림 대상 작업 식별자입니다.

|``subject.name``
|작업 제목 스냅샷
|알림 원인이 된 시점의 작업 제목입니다.

|``link.type``
|``TASK_DETAIL``
|클라이언트가 작업 상세 화면 이동을 구성할 수 있는 링크 힌트입니다.

|``link.workspaceId``
|``workspaceId``
|작업 상세 화면 진입에 사용할 워크스페이스 식별자입니다. 작업 식별자는 ``subject.id``를 사용합니다.

|``changes[].from``
|변경 전 값
|변경 요소에 따라 문자열, 숫자, Long Unix epoch seconds, 객체 또는 ``null``이 내려올 수 있습니다.

|``changes[].to``
|변경 후 값
|변경 요소에 따라 문자열, 숫자, Long Unix epoch seconds, 객체 또는 ``null``이 내려올 수 있습니다.
|===

== 변경값 타입

[cols="1,1,4",options="header",role="api-table"]
|===
|element |값 타입 |설명

|``STATUS``
|String
|``TODO``, ``IN_PROGRESS``, ``DONE`` 같은 작업 상태 값입니다.

|``PROGRESS``
|Number
|0부터 100까지의 작업 진행도 값입니다.

|``START_DATE``
|Number 또는 ``null``
|작업 시작 예정일입니다. 값이 있으면 Long Unix epoch seconds 기준입니다.

|``DUE_DATE``
|Number 또는 ``null``
|작업 마감 예정일입니다. 값이 있으면 Long Unix epoch seconds 기준입니다.

|``ASSIGNEE``
|Object 또는 ``null``
|담당자 정보입니다. 객체 값은 ``memberId``, ``userId``, ``displayName``을 포함합니다.
|===

== 복합 변경 분리 기준

하나의 작업 수정에서 여러 요소가 동시에 변경될 수 있습니다. 서버는 같은 활동 레코드에서 파생된 알림을 의미 단위로 분리하고, 각 알림의 ``changes``에는 해당 알림 유형에 필요한 요소만 포함합니다.
단, ``TASK_DATE_CHANGED``와 ``TASK_PROGRESS_CHANGED``는 노이즈 감소 정책에 따라 현재 알림 기록과 WebSocket payload를 생성하지 않습니다.

[cols="1,2,3",options="header",role="api-table"]
|===
|원본 변경 요소 |발송 알림 순서 |알림별 ``changes`` 포함 요소

|``ASSIGNEE``, ``DUE_DATE``
|``TASK_UNASSIGNED`` -> ``TASK_ASSIGNED``
|``TASK_UNASSIGNED``/``TASK_ASSIGNED``는 ``ASSIGNEE``만 포함합니다. ``DUE_DATE`` 변경은 현재 알림으로 생성하지 않습니다.

|``STATUS``, ``DUE_DATE``
|``TASK_STATUS_CHANGED``
|``TASK_STATUS_CHANGED``는 ``STATUS``만 포함합니다. ``DUE_DATE`` 변경은 현재 알림으로 생성하지 않습니다.

|``PROGRESS``, ``START_DATE``
|없음
|``START_DATE``와 ``PROGRESS`` 변경 알림은 현재 생성하지 않습니다.
|===

서버 내부에서는 같은 활동 레코드에서 파생된 알림을 그룹으로 다루고, 아래 우선순서로 저장 및 발송합니다. 이 순서 값은 응답 payload에 노출되지 않으며, 클라이언트는 수신 순서와 목록 순서를 그대로 사용하면 됩니다.

[cols="1,1,4",options="header",role="api-table"]
|===
|순서 |알림 유형 |의미

|1
|``TASK_CREATED``
|작업 생성 사실을 먼저 알립니다.

|2
|``TASK_UNASSIGNED``
|직전 담당자가 해제되었음을 알립니다.

|3
|``TASK_ASSIGNED``
|새 담당자로 지정되었음을 알립니다.

|4
|``TASK_DATE_CHANGED``
|예약 순서입니다. 현재는 알림으로 생성하지 않습니다.

|5
|``TASK_STATUS_CHANGED``
|작업 상태 변경을 알립니다.

|6
|``TASK_PROGRESS_CHANGED``
|예약 순서입니다. 현재는 알림으로 생성하지 않습니다.
|===

.복합 변경 분리 예시
[source,json]
----
{
  "sourceChanges": [
    {
      "element": "ASSIGNEE",
      "from": null,
      "to": {
        "memberId": 55,
        "userId": 78,
        "displayName": "Target"
      }
    },
    {
      "element": "DUE_DATE",
      "from": 1781222400,
      "to": 1781827200
    }
  ],
  "notifications": [
    {
      "type": "TASK_ASSIGNED",
      "changes": [
        {
          "element": "ASSIGNEE",
          "from": null,
          "to": {
            "memberId": 55,
            "userId": 78,
            "displayName": "Target"
          }
        }
      ]
    }
  ]
}
----

== 클라이언트 처리 기준

[cols="1,4",role="endpoint-summary"]
|===
|문장 조립 |``type``, ``actor``, ``subject``, ``scope``를 기준으로 클라이언트에서 문장을 조립합니다.
|대표 프로필 |프로필 이미지는 ``profile``을 사용합니다. ``TASK_ASSIGNED``와 ``TASK_UNASSIGNED``는 ``RECIPIENT``, 그 외 작업 변경 알림은 ``ACTOR`` 기준으로 내려줍니다. 이미지가 ``null``이면 link:notifications-enum-reference.html#notification-profile-source-type[NotificationProfileSourceType]을 기본 아바타 문자 선택 기준으로 사용합니다.
|날짜/진척도 변경 |``TASK_DATE_CHANGED``와 ``TASK_PROGRESS_CHANGED``는 예약 타입이지만 현재 알림 기록과 WebSocket payload를 생성하지 않습니다.
|생성/할당 순서 |작업 생성과 동시에 담당자로 지정된 사용자는 ``TASK_CREATED``를 먼저 받고 ``TASK_ASSIGNED``를 이어 받을 수 있습니다.
|담당자 변경 분기 |``TASK_ASSIGNED``는 새 담당자 관점, ``TASK_UNASSIGNED``는 직전 담당자 관점으로 표시합니다.
|상세 diff |각 알림의 ``changes``는 해당 ``type``에 필요한 요소만 포함합니다. 클라이언트는 별도 element 필터링 없이 ``type``별 상세 diff를 표시할 수 있습니다.
|링크 처리 |``link.type``이 ``TASK_DETAIL``이면 ``link.workspaceId``와 ``subject.id``를 사용해 작업 상세 화면으로 이동할 수 있습니다.
|상태 동기화 |작업 상세 화면에 있는 사용자는 알림 수신 후 link:workspaces-tasks-get.html[작업 상세 조회 API]를 다시 호출해 서버 상태와 동기화할 수 있습니다.
|fallback |알 수 없는 작업 알림 유형을 수신하면 데이터를 버리지 말고 기본 작업 알림 문구로 표시한 뒤 상세 화면 재조회로 보정하는 것을 권장합니다.
|===
